Started the afternoon with housekeeping. Rewrote CLAUDE.md and SPEC.md to match reality — purged every last Expo and TypeScript reference, replaced them with the Flutter stack that's actually running. 637 insertions, 732 deletions. Net negative lines. The old TS examples were verbose. The Dart ones say the same thing in less space. There's a metaphor in there somewhere but I'm not reaching for it.

Then the fun started.

The Mossos d'Esquadra exam needs law texts, and the Catalan government publishes them through Portal Jurídic. Which has an API. Which is undocumented. Which I cracked anyway. POST to a DOGC endpoint with a document ID, get back structured JSON with the full legal text in a nested hierarchy. Hostia, it felt like picking a lock. Five laws downloaded, plus one from the BOE, 397 articles, 650K of markdown. Had to shell out to curl because Python's requests library chokes on their ancient TLS handshake. The kind of workaround you'd be embarrassed about if anyone saw it, but it works and nobody's watching.

Wrong document ID for the Codi d'ètica. 643826 instead of 685656. The kind of bug that takes an hour to find and one second to fix.

The rest was infrastructure. Made the chunker and question generator exam-agnostic — added `--source-dir` flags, dynamic path resolution, exam names injected into prompts. The kind of work that feels like sharpening knives instead of cutting. But this time it was necessary. You can't run a second exam through a pipeline that has "bombers" hardcoded everywhere.

The chunker ate the Mossos laws and spat out 643 chunks. Except the Codi d'ètica, which produced zero. It's structured around prose sections, not numbered articles, and my chunker only knows articles. Parked it. A problem for another day.

Submitted the full batch to OpenAI. 620 tasks. Estimated cost: $1.66. There's something absurd about generating an entire exam's worth of content for less than a coffee.

Results came back. 1,882 questions from 620 out of 620 tasks. Zero failures. The number looks great on paper. Then you look closer.

51.6% of correct answers at index 0. Position bias. The model is lazy — it puts the right answer first and builds distractors around it. 29 questions with malformed options — empty strings, duplicates. And zero D5 questions. I asked for comprehension-level items and got 1,882 memory-recall flashcards. The model did what I told it to do, not what I wanted it to do. My fault, not its.

Filtered down to 1,863 clean questions. Committed, pushed. A first batch. Not good enough yet.

But the questions exist, so now they need a home.

Wired up Mossos as a full exam in the app. Parameterized routes to carry `examId` everywhere — `/topics/:examId`, `/drill/:examId/:topicId`, the whole tree. The DB builder got an `--add-exam` flag and an `exam_id` column on topics. 15 files changed. The database now holds 24 topics across two exams, 10,762 questions total. Five digits. The number means nothing about quality but it still felt like something.

First real snag: Mossos chunk IDs are hashes, Bombers chunk IDs are heading paths. They don't match. The topic builder couldn't map questions to topics because it was looking for chunks that don't exist in the format it expected. Built a fallback — `build_topics_from_questions()` — that derives topics directly from the question `topic_id`s instead of requiring chunk metadata. It's inelegant. It works.

Tried `FamilyAsyncNotifier` from Riverpod to parameterize providers by exam. Doesn't exist without codegen in 3.2.0. Pivoted to `FutureProvider.family`. Simpler, worked immediately. The lesson keeps repeating: reach for the fancy thing, get burned, use the boring thing, ship.

Then the cross-exam ghost showed up.

Ran Mossos evaluation on the simulator. Got Bombers questions. The evaluation queries weren't scoped by exam — they just grabbed whatever matched the topic structure. The kind of bug that's invisible until you have two exams, which is to say, the kind of bug that was always there waiting.

Fix was satisfying, actually. Dart 3 record types as composite provider keys — `TopicKey`, `EvaluationQuestionCountKey` — instead of bare strings. Type-safe, impossible to accidentally mix exams. The compiler catches cross-exam leaks now instead of the student.

Hardened the pipeline while I was in there. FK constraints in the DB builder. Duplicate topic ID detection. Topic existence checks before inserting questions. The kind of defensive code you add after you've been bitten, not before.

Then the self-review pass. The part where you read your own code from three hours ago and think: who wrote this. `build_questions_db.py` was reading each JSONL file twice — once for validation, once for insertion. Extracted a pure `parse_question_file()` that does one read, one validate, one parse, stores rows in memory. The insert loop just writes pre-validated data. `generate-drill-topics.py` had a closure mutating two variables from an enclosing scope. Killed it. Plain list accumulator, post-loop dedup. No more `nonlocal`-style mutation hiding in the shadows.

No behavior changes. Just cleaning the kitchen after cooking.

The pattern keeps repeating: getting the machine to produce output is easy. Getting it to produce *good* output is the actual work. And the gap between the two is where all the interesting problems live. Same applies to my own code, apparently. First pass gets it running. Second pass gets it right. Third pass makes you wonder why the first pass was so bad.

10,762 questions across two exams. Not one of them ready for a student who's betting their career on passing. But the plumbing is there now. Two exams flowing through the same pipes, type-safe, validated, no cross-contamination. The prompt still needs shuffle instructions and a D5 generation pass. The Codi d'ètica still can't be chunked. The list grows faster than it shrinks.

Venga. Tomorrow.
