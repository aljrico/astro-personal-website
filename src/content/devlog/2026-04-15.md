Started with the profiler. Twenty seconds, iPhone 17 sim, Time Profiler doing its thing. Had to use the sim because xctrace is currently broken against physical devices — some regression where it just hangs on "Waiting for device to boot" even when the device is clearly awake and online. CLI, Instruments GUI, both affected. Not my bug to fix, but still. Nothing like spending twenty minutes debugging your debugging tools.

Anyway. The sim trace came back and there it was: `PremiumSkinIconView`, sitting at the top of the app-code hotspot list like it owned the place. Thirteen percent CPU. Every click.

I had a theory about why. I had the right general area. Then I ran it by the AI I've been pairing with and it came back with a tighter answer than mine. Sharper. More complete. Not wrong-theory-corrected — more like I had the sketch and it handed me the blueprint.

I'm still processing how I feel about that. Not bad, exactly. More like: *oh, so this is what that looks like from the outside.* The tool saw the invalidation chain more clearly because it wasn't anchored to my mental model of how the code was supposed to work. I was debugging my intentions. It was debugging the code. Different problem.

The actual culprit: `arePurchasesAvailable` reading click counts and wealth from the game state, which meant every single click was blowing up the view. And `currentSale` called three times per body pass, each one reshuffling the weighted candidate list from scratch. Not once. *Three times.* Independently. On every frame.

You know that feeling when you see the bug and it's almost impressive? Like, this thing has been sitting here this whole time, quietly doing thousands of unnecessary calculations, and the app was just... absorbing it? Yeah. That.

The fix: memoised `currentSale` properly — keyed on rotation index, month, owned skins. Cache the result. Main queue, no eviction. Also caught a latent bug while we were in there: `availability.start` was silently using call time instead of the rotation bucket start. Harmless in practice because the only reader filters upcoming hardcoded sales where it would never have matched anyway, but — still. A bug is a bug even when it doesn't hurt anyone yet.

Shipped it. Post-fix trace: skin-icon cluster went from 500ms to 12ms.

Ninety-seven percent.

I keep looking at that number. That's not a fix, that's an execution. That's the kind of result you screenshot.

The workflow thing is worth noting too, because it surprised me. Parallel human+AI on the same problem is messy if you both start coding immediately — you diverge fast and then spend more time reconciling than you saved. But force a written spec first — agree on *what* you're building before either of you touches a file — and it collapses cleanly. The spec is the shared memory. Without it you're both just vibing and calling it collaboration.

One thing that didn't work: the profiler script that's supposed to parse trace output has a bug where it doesn't resolve `<frame ref=>` pointers. So about thirty percent of the leaf nodes showed up as "unknown" in the output. Had to write a one-off workaround inline, which worked but felt sloppy. Worth fixing upstream properly at some point. Probably won't though. Will absolutely come back to this and say "why did past me leave this broken."

There's still more in the profile. The GameState update is slow and I know where to look next. But I'm not touching it tonight. The 97% drop is enough. Sometimes you have to let the win be the win instead of immediately sprinting to the next problem.

Tomorrow: break the invalidation chain at the source. Until then — the explosions can wait.
