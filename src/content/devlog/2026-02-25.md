Started the day on Hollows. Shipped four node types in a single commit — toll, shrine, sanity drain, mimic. The mimic disguises as a chest then attacks you. That's the kind of design that writes itself. The shrine trades fangs for sanity and it *feels* right in a way I can't explain mechanically. Swapped all the SF Symbol icons for the Rhos fantasy pack and spent way too long picking the boss icon. Rejected the king. Went with an eye silhouette. Lovecraft over Tolkien. The eye breathes purple at 1.5x scale and I keep opening the app just to look at it.

Then switched to FlamaCoach and fell into the description trap.

Twenty topic descriptions. That's it. Twenty one-liners for the Mossos Guide drill cards. Should take, what, an hour? I'll batch it through the API.

Round one: gpt-4.1-nano. Cheap, fast, terrible. Two-sentence summaries that read like Wikipedia intros. Nobody's reading that on a phone card. Upgraded to gpt-5.2 with low reasoning effort. Better, but still wordy. $0.012 for twenty descriptions that I immediately wanted to rewrite.

Round two: tighter prompt. "UNA SOLA FRASE, màxim 20 paraules, estil telegràfic." Got keyword soup back. Technically compliant, emotionally dead. The kind of text that makes you skip past it without reading. Removed the maxLines clamp from the cards so they'd grow dynamically, which was the right fix for the UI but didn't fix the words.

Round three: I wrote them myself. All twenty. By hand.

The first one set the tone — "Evolució política i social de Catalunya des de l'antiguitat fins al segle XVIII: romanització, feudalisme, expansió mediterrània i guerres." Framing clause, colon, key milestones. Rhythm. I did the other nineteen in that voice and they were done in maybe forty minutes.

Three iterations, two batch API calls, ~$0.024 in tokens, and the answer was always just: write them yourself, idiota. Twenty items where tone matters isn't a scale problem. It's a craft problem. The batch script lives on for when I need 200 descriptions. For 20 high-visibility lines that set the personality of the app, the LLM was the wrong tool.

There's a lesson I keep relearning. The instinct to automate isn't always about efficiency. Sometimes it's about avoiding the blank page. Easier to tweak a prompt than to stare at an empty string and decide what the description *should* say. The batch script is a shield against having to make creative choices. And creative choices are exactly what twenty topic descriptions need.

Fixed a hot restart bug along the way. The database was "disappearing" after hot restart — turns out the MD5 hash changes when you rebuild the asset, triggers a re-copy, but hot restart doesn't re-push assets so the static DB reference goes null. Moved `DatabaseService.initialize()` from the splash screen to `main()`. Five-minute fix once I understood it. Felt like an hour before I did.

Wrapped the descriptions work by documenting the whole pipeline in the README. Chunk → questions → drill topics → descriptions → DB. Included the style examples — good vs bad descriptions, inline, while I still remembered why the bad ones were bad. Documentation written the same day as the decisions is a completely different artifact than documentation written a month later. One is a trail. The other is archaeology.

Lunchtime turned into a code review session. Six issues triaged on the Flutter app, four actually needed work. Two were false alarms — the const constructors and hardcoded colors looked wrong in isolation but were fine in context. Knowing when not to fix something is its own skill.

The real win was extracting the question UI. Drill screen and evaluation screen had been copy-pasting the same option buttons, answer feedback, continue buttons — about 240 lines of near-identical widgets living in two places. Pulled them into a shared `question_widgets.dart`. The interesting merge decision was the ContinueButton: drill had an `isDisabled` flag, evaluation had `isLastQuestion`. Stared at both for a minute, realized `onTap: null` already handles disabled state, so `isLastQuestion` with a default of false covers both cases. One param instead of two. Net result across eight files: added 279 lines, deleted 507. That ratio feels good.

Found a real bug in there too. The dark mode was flickering on launch — light theme flash, then snap to dark. ThemeModeNotifier's `build()` was firing `_loadFromPrefs()` async without awaiting it. Same pattern as the hot restart bug from the morning: initialization that runs too late. Used the same fix — static `preload()` called in `main()` before `runApp()`. Two initialization bugs in one day, same root cause, same solution. Patterns.

Also snuck in a `difficultyColor(int)` extension on AppColors. Five difficulty levels, five colors, duplicated across who knows how many widgets. Now it's one lookup in theme.dart. The kind of thing that takes three minutes and prevents three future bugs.

Afternoon was pipeline hygiene. Code-reviewed the length-cue fixer — a big script that detects when one MCQ option is suspiciously longer than the others and pads or trims the rest to remove the tell. Test-taking 101: if answer C is twice as long as A, B, and D, you pick C. The script catches that and fixes it, with 34 hand-tagged manual overrides for the cases where the algorithm can't decide.

The review was the satisfying kind. Four surgical fixes: `python3` → `uv run`, mutable `list` returns → `tuple`, and the one that mattered — the frozen dataclass had a `raw: dict` field dangling there like a mutable time bomb. Replaced it with explicit `question_id` and `correct_index` fields. The kind of cleanup where every change makes the code more honest about what it actually needs.

Twenty-six kilobytes of script to solve a problem most people don't even notice they have. But if you're studying for an exam and three of the four options are eight words and one is twenty-two, your brain notices before you do. The bias is invisible and powerful. Removing it is invisible too. Nobody will ever think "wow, these options are well-balanced." They'll just stop guessing right for the wrong reasons.

Late afternoon I went full UI polish mode. Home screen first: swapped the fire emoji for a proper Lucide flame icon, restructured the layout so the red CTA button pins to the bottom instead of scrolling away, simplified the button text to just "ESTUDIA" in bold. No topic name, no arrow icon. Just the verb. Decoupled it from the async topics provider too — the button is tappable instantly now, navigation handles the loading. Perceived responsiveness matters more than actual responsiveness.

Topics screen got the same treatment. Iterated on the back-button icon three times — started with home, tried arrowBigLeft, landed on doorClosed. Which is weird and I like it. Added "Temari" as a prefix to exam names, dialed the title font down from headlineLarge to headlineMedium. Small moves that shift the whole feel.

Last thing: the Senyera stripe. The hero card had a Catalan flag stripe across it, built with a LinearGradient where the stops were a mess — the red bands were 12% wide, the amber bands 11%, nothing matched. Ripped out the gradient entirely and replaced it with a Row of nine Expanded ColoredBox children. Equal width. Done. Moved it from bottom to top, then bottom, then top again. Top won. Doubled the height from 4 to 8 pixels. Wrapped the whole card in a ClipRRect so the stripe edges clip to the card's border radius — the flat top edges blend into the content and the card handles the corner rounding. The kind of fix where you delete a gradient config block and replace it with something a first-year CS student could read.

Three projects. One day. The Hollows morning was pure momentum — aesthetic instinct, ship it. The FlamaCoach descriptions were the humbling middle — circling the automation, realizing the machine couldn't do what I needed, then doing it faster than I spent prompting. The code review was craft — extracting, unifying, tightening. And the afternoon was the mode I love most: making things *look* right. Icon by icon, pixel by pixel, until the screen feels like it belongs to a real app and not a prototype. All four modes valid. All four mine.
