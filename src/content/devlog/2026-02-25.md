Started the day on Hollows. Shipped four node types in a single commit — toll, shrine, sanity drain, mimic. The mimic disguises as a chest then attacks you. That's the kind of design that writes itself. The shrine trades fangs for sanity and it *feels* right in a way I can't explain mechanically. Swapped all the SF Symbol icons for the Rhos fantasy pack and spent way too long picking the boss icon. Rejected the king. Went with an eye silhouette. Lovecraft over Tolkien. The eye breathes purple at 1.5x scale and I keep opening the app just to look at it.

Then switched to FlamaCoach and fell into the description trap.

Twenty topic descriptions. That's it. Twenty one-liners for the Mossos Guide drill cards. Should take, what, an hour? I'll batch it through the API.

Round one: gpt-4.1-nano. Cheap, fast, terrible. Two-sentence summaries that read like Wikipedia intros. Nobody's reading that on a phone card. Upgraded to gpt-5.2 with low reasoning effort. Better, but still wordy. $0.012 for twenty descriptions that I immediately wanted to rewrite.

Round two: tighter prompt. "UNA SOLA FRASE, màxim 20 paraules, estil telegràfic." Got keyword soup back. Technically compliant, emotionally dead. The kind of text that makes you skip past it without reading. Removed the maxLines clamp from the cards so they'd grow dynamically, which was the right fix for the UI but didn't fix the words.

Round three: I wrote them myself. All twenty. By hand.

The first one set the tone — "Evolució política i social de Catalunya des de l'antiguitat fins al segle XVIII: romanització, feudalisme, expansió mediterrània i guerres." Framing clause, colon, key milestones. Rhythm. I did the other nineteen in that voice and they were done in maybe forty minutes.

Three iterations, two batch API calls, ~$0.024 in tokens, and the answer was always just: write them yourself, idiota. Twenty items where tone matters isn't a scale problem. It's a craft problem. The batch script lives on for when I need 200 descriptions. For 20 high-visibility lines that set the personality of the app, the LLM was the wrong tool.

There's a lesson I keep relearning. The instinct to automate isn't always about efficiency. Sometimes it's about avoiding the blank page. Easier to tweak a prompt than to stare at an empty string and decide what the description *should* say. The batch script is a shield against having to make creative choices. And creative choices are exactly what twenty topic descriptions need.

Fixed a hot restart bug along the way. The database was "disappearing" after hot restart — turns out the MD5 hash changes when you rebuild the asset, triggers a re-copy, but hot restart doesn't re-push assets so the static DB reference goes null. Moved `DatabaseService.initialize()` from the splash screen to `main()`. Five-minute fix once I understood it. Felt like an hour before I did.

Wrapped the descriptions work by documenting the whole pipeline in the README. Chunk → questions → drill topics → descriptions → DB. Included the style examples — good vs bad descriptions, inline, while I still remembered why the bad ones were bad. Documentation written the same day as the decisions is a completely different artifact than documentation written a month later. One is a trail. The other is archaeology.

Afternoon started with pipeline hygiene. Code-reviewed the length-cue fixer — a big script that detects when one MCQ option is suspiciously longer than the others and pads or trims the rest to remove the tell. Test-taking 101: if answer C is twice as long as A, B, and D, you pick C. The script catches that and fixes it, with 34 hand-tagged manual overrides for the cases where the algorithm can't decide.

The review was the satisfying kind. Four surgical fixes: `python3` → `uv run`, mutable `list` returns → `tuple`, and the one that mattered — the frozen dataclass had a `raw: dict` field dangling there like a mutable time bomb. Replaced it with explicit `question_id` and `correct_index` fields. The kind of cleanup where every change makes the code more honest about what it actually needs.

Twenty-six kilobytes of script to solve a problem most people don't even notice they have. But if you're studying for an exam and three of the four options are eight words and one is twenty-two, your brain notices before you do. The bias is invisible and powerful. Removing it is invisible too. Nobody will ever think "wow, these options are well-balanced." They'll just stop guessing right for the wrong reasons.

Then the pipeline review turned into a Flutter review. Six issues triaged from the code review, four actually needed work. Two were false alarms — the "add const constructors" complaint dissolved on inspection, and the "hardcoded colors" flag was pointing at colors that were already centralized. Knowing when *not* to fix something is half of code review.

The real win was the widget extraction. Drill screen and evaluation screen had been sharing the same question UI through copy-paste — option buttons, answer feedback, the continue button, the whole footer. About 240 lines duplicated across two files. Pulled them all into `question_widgets.dart` and the merge decisions were more interesting than expected. Both screens had a ContinueButton but with different APIs — drill used `isDisabled`, evaluation used `isLastQuestion`. Collapsed both into a single `isLastQuestion` param with a default of false, because `onTap: null` already handled the disabled state. The old API was modeling two concepts where one existed.

Net result: +279 −507 lines across eight files. Negative line counts are the best line counts.

While I was in the notifiers I caught a real bug. `ThemeModeNotifier.build()` was firing an async `_loadFromPrefs()` without await — so the widget tree would render with the default light theme, then the prefs would load, then it would snap to dark. Visible flicker on every cold start. Used the same static preload pattern as the database init: call it in `main()` before `runApp()`, let the notifier read from the already-loaded value synchronously. Same fix shape as this morning's hot restart bug. Boundaries want to be initialized early and read late.

Also added a `difficultyColor()` extension on AppColors. Five difficulty levels, five colors, duplicated in three widgets. Now it's one method. Small, but the kind of thing that rots if you leave it.

Late afternoon shifted to visual polish. The home screen got a layout restructure — fire emoji replaced with a proper Lucide flame icon, the red CTA button pinned to the bottom of the screen instead of floating inside the scroll view. Simplified the button text to just "ESTUDIA" in bold. No topic name, no arrow icon, no cleverness. The button is always tappable now, even while the topics provider is still loading, because I decoupled it from the async state. Perceived responsiveness matters more than actual responsiveness. If the button looks dead for 200ms while data loads, that's 200ms where the user thinks the app is broken.

Topics screen got smaller tweaks — iterated on the back-button icon through three options. Home icon felt wrong because you weren't going home, you were leaving. Arrow felt generic. Settled on a door icon, which is either poetic or ridiculous depending on your mood. Added "Temari" as a prefix to exam names and pulled the title font down a size. The kind of decisions that take five seconds to make and thirty minutes to stop second-guessing.

Last thing: the Senyera stripe. The hero card had a Catalan flag stripe across it rendered with a LinearGradient, and the stops were a disaster — manually positioned, red bands twice as wide as the amber ones, nothing actually even. Ripped it out and replaced it with a Row of nine Expanded ColoredBox children. Equal width. Guaranteed. Moved it from bottom to top, then back to bottom, then back to top. Top works as a header banner. Doubled the height from 4px to 8px. Wrapped the whole card in a ClipRRect so the stripe edges clip to the border radius naturally instead of poking out with sharp corners.

Nine equal stripes. It shouldn't matter this much. It does.

Two projects. One day. Five modes. The Hollows work was all momentum — four node types, aesthetic decisions, ship it. The description work was circling, iterating, realizing the machine couldn't do what I needed. The code review was surgical — finding the seams where duplication hid and pulling them clean. The UI polish was the flow state, the thing that keeps me opening the editor. And the Senyera stripe was the obsessive last-mile work that nobody notices and everybody feels. All five modes valid. All five mine. Minus 507 lines and the app is better for it.
